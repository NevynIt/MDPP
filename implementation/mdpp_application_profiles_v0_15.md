This document describes three target applications and the components/plugins needed to implement them.

---

## 1. Target A: Node.js CLI renderer

### 1.1. Goal

A Node.js command-line application reads an md++ entry document and produces HTML or PDF.

Example commands:

```bash
mdpp render ./docs/report.md --out ./dist/report.html
mdpp render ./docs/report.md --pdf --out ./dist/report.pdf
mdpp validate ./docs/report.md --format json
mdpp bundle ./docs/report.md --out ./dist/report.md.zip
```

### 1.2. Required capabilities

The CLI should support:

- local filesystem repositories;
- nested includes;
- repository-qualified resources;
- theme/layout/stylesheet resources;
- inline and external model parsing and model rendering;
- HTML output;
- PDF output;
- structured diagnostics;
- dependency reporting;
- optional watch mode.

### 1.3. Required components

| Component | Purpose |
|---|---|
| `@mdpp/cli` | CLI shell and argument handling. |
| `@mdpp/processor` | Main processing pipeline. |
| `@mdpp/repository-file` | Local file access. |
| `@mdpp/plugin-registry` | Plugin discovery and dispatch. |
| `@mdpp/render-core` | Render tree generation. |
| `@mdpp/export-html` | HTML serialization. |
| `@mdpp/export-pdf` | PDF generation. |
| `@mdpp/diagnostics` | CLI and JSON diagnostics. |
| `@mdpp/source-map` | Source map generation. |

### 1.4. Required plugins

Use the Node CLI HTML or PDF bundle from the plugin catalog.

### 1.5. CLI outputs

Recommended output modes:

| Mode | Output |
|---|---|
| `--out file.html` | Standalone HTML file or HTML fragment, according to option. |
| `--pdf --out file.pdf` | PDF file. |
| `--diagnostics json` | Machine-readable diagnostics. |
| `--dependencies json` | Used resources and plugins. |
| `--source-map json` | Source map. |
| `--bundle file.zip` | Portable resource bundle. |

### 1.6. Non-goals for first CLI release

- full visual editor behavior;
- arbitrary remote resource access by default;
- live main-thread interaction execution in exported PDF;
- guaranteed pixel-identical PDF across all operating systems unless the export backend and fonts are pinned.

## 2. Target B: React viewer running from `file://`

### 2.1. Goal

A React component displays md++ content in a browser application that may be opened directly from `file://`. It supports interactions and dynamic content updates.

Example usage:

```tsx
<MdppViewer
  entry="main.md"
  resources={resourceMap}
  plugins={plugins}
  onDiagnostics={setDiagnostics}
  onInteraction={handleInteraction}
/>
```

### 2.2. Important browser constraint

A viewer loaded from `file://` should not assume it can fetch arbitrary sibling files by path. Browser behavior differs and access is restricted.

The viewer should therefore support at least one of these input modes:

- source text provided directly as a prop;
- resource map provided as a prop;
- drag/drop or file picker loaded resources;
- prebuilt md++ bundle;
- embedded static resources generated by the CLI.

### 2.3. Required capabilities

The viewer should support:

- in-memory and bundle repositories;
- render snapshot mounting;
- DOM patch updates;
- interaction bindings;
- worker-compatible rendering;
- diagnostics callbacks;
- source map callbacks;
- dynamic resource replacement.

### 2.4. Required components

| Component | Purpose |
|---|---|
| `@mdpp/react-viewer` | React component API. |
| `@mdpp/dom-adapter` | Mount snapshots and apply patches. |
| `@mdpp/processor` | Parse/resolve/render pipeline. |
| `@mdpp/repository-memory` | Resource map access. |
| `@mdpp/repository-bundle` | Bundle access. |
| `@mdpp/plugin-runner` | Worker/main-thread plugin calls. |
| `@mdpp/interactions` | Event normalization and action dispatch. |
| `@mdpp/source-map` | Node/source mapping. |

### 2.5. Required plugins

Use the React `file://` viewer bundle from the plugin catalog.

### 2.6. Dynamic update model

The viewer should expose update inputs such as:

```ts
interface MdppViewerUpdate {
  ref: string;
  kind: "whole" | "text-diff" | "binary" | "deleted";
  text?: string;
  content?: ArrayBuffer;
}
```

The viewer passes updates to the renderer. The renderer returns patches, source map changes, interaction changes, diagnostics, resources, plugins, and updated state.

### 2.7. Non-goals for first viewer release

- arbitrary local filesystem browsing from `file://`;
- full WYSIWYG editing;
- untrusted plugin execution without host policy decisions;
- all PDF pagination behavior in the browser viewer.

## 3. Target C: Visual editor

### 3.1. Goal

A visual editor allows users to author md++, inspect rendered structure, navigate diagnostics, and edit source with rendered feedback.

This target should be considered experimental. The current specs are close to supporting it, but the editor will stress source maps, patch stability, reversible transformations, diagnostics, and plugin metadata.

### 3.2. Recommended first editor shape

The first editor should not attempt full WYSIWYG editing.

Recommended first version:

- source editor on the left;
- live rendered preview on the right;
- diagnostics panel;
- source-to-render and render-to-source selection;
- block inspector;
- layout/page/area inspector;
- plugin trace view;
- quick fixes for stable diagnostic codes.

### 3.3. Required components

| Component | Purpose |
|---|---|
| `@mdpp/editor-core` | Editor-neutral state and services. |
| `@mdpp/editor-react` | Optional React shell. |
| `@mdpp/react-viewer` | Rendered preview. |
| `@mdpp/source-map` | Selection mapping. |
| `@mdpp/diagnostics` | Diagnostic display and code lookup. |
| `@mdpp/plugin-registry` | Plugin metadata and trace display. |
| `@mdpp/repository-file` or workspace provider | Project resource access. |
| `@mdpp/repository-memory` | Unsaved edits and virtual documents. |

### 3.4. Required plugins

Start from the React viewer bundle and add the editor support plugins from the plugin catalog.

### 3.5. Additional standardization needed before full editor

The editor target needs further standardization around:

- stable document artifact lifecycle;
- reversible source transformations;
- precise source ranges for generated/plugin output;
- block-level editing operations;
- plugin-provided inspectors;
- schema-driven attribute completion;
- quick-fix action format;
- undo/redo interaction with render updates.

Until then, visual editing should be limited to source editing plus rich preview and inspection.

## 4. Shared host profile matrix

| Concern | Node CLI | React `file://` viewer | Visual editor |
|---|---:|---:|---:|
| Local filesystem read | yes | no, except user-selected files | yes, through host/workspace |
| Local filesystem write | optional | no | yes, through host/workspace |
| Memory repository | yes, tests | yes | yes |
| Bundle repository | yes, export/import | yes | yes |
| Worker execution | optional | recommended | recommended |
| JSON-RPC process plugins | optional | uncommon | optional |
| DOM adapter | no for static HTML; yes for preview tests | yes | yes |
| PDF export | yes | optional/no | optional |
| Dynamic updates | watch mode | yes | yes |
| Interactions | exported HTML only | yes | yes |
| Source maps | yes | yes | required |
| Diagnostics | required | required | required |

---

## 5. Recommended implementation order

1. Shared types, schemas, and diagnostics.
2. GFM parser integration and directive collection.
3. Repository core plus file and memory repositories.
4. Include resolver and requirement resolver.
5. Basic HTML renderer and source maps.
6. CLI HTML output.
7. Plugin registry and plugin manifest validation.
8. Mermaid, math, syntax highlight, DOT inline/external model and render plugins.
9. Layout interpretation and basic area renderers.
10. CLI PDF output.
11. Bundle export/import.
12. React viewer with DOM adapter and interaction dispatch.
13. Dynamic update and patch stabilization.
14. Editor core prototype.

## 6. Target D: Office-to-md++ normalization pipeline

### 6.1. Goal

A conversion pipeline reads DOCX/PPTX or similar Office-like source documents and produces semantic md++ that can be edited as text and rendered to HTML/PDF.

```text
DOCX/PPTX -> md++ source + sidecar metadata + diagnostics -> text editing -> HTML/PDF rendering
```

### 6.2. Required capabilities

The pipeline should support:

- headings and paragraphs as Markdown;
- named Word/PowerPoint styles as md++ classes;
- tables as Markdown tables when simple;
- images as exported assets referenced by Markdown image syntax;
- headers, footers, and page numbers as page furniture;
- comments, speaker notes, and review metadata as sidecar artifacts;
- lossy import diagnostics for unsupported or simplified source features.

### 6.3. Required components

| Component | Purpose |
|---|---|
| `@mdpp/import-office` | Convert Office-like sources to semantic md++ and sidecars. |
| `@mdpp/presentation-resolver` | Resolve generated classes, theme declarations, and page furniture. |
| `@mdpp/diagnostics` | Emit stable lossy-import diagnostics. |
| `@mdpp/schemas` | Validate generated artifacts and sidecars. |
